## Pre-Written Response Angles

### Angle 1: Generous, Identity-Focused
```
The pattern that emerged most clearly: 63% of engineers taking the quiz said 
they felt like "middlemen" in their own code — reviewing AI-generated work, 
not writing it. 58% reported measurable skill decline.

This isn't laziness or resistance to change. It's a real cognitive shift 
from using tools that bypass productive struggle — which is where 
actual learning happens.

Happy to answer questions about the research behind this.
```

### Angle 2: On Junior Engineers
```
The part that worries me most: the junior engineers who have never shipped 
code without AI. They're building on a foundation where the craft satisfaction, 
the "I built this" feeling, was never fully established.

When something goes wrong in a way AI didn't anticipate — and it will — 
they won't have the instincts to fall back on. Because the instincts 
come from the struggle, and the struggle was AI-assisted from day one.

The clearing-ai.com/junior-engineers.html page has more on this.
```

### Angle 3: On "Just Take Breaks" Rebuttal
```
The common response to this is "just take breaks" or "set boundaries."

AI Fatigue isn't primarily about overwork. It's about a structural change 
in how expertise accumulates. When you use AI tools heavily, you learn 
differently — not necessarily worse in the short term, but with a thinner 
mental model that shows up in unexpected ways.

The recovery isn't just rest. It's intentional practice that rebuilds 
the conditions expertise needs to develop.

(And yes, boundaries help. They're necessary but not sufficient.)
```

### Angle 4: On the Research Basis
```
This draws on established cognitive science: cognitive load theory (Sweller), 
attention residue (Leroy), automation bias (Parasuraman), 
flow state disruption (Csikszentmihalyi).

The AI fatigue phenomenon is real — what's new is the scale and the 
speed of the shift. The research has been waiting for this application.
```

### Angle 5: On Not Being Anti-AI
```
I'm not arguing against AI tools. I use them.

The argument is about the *design* of how we integrate them — 
that matters as much as the tools themselves.

Tools that bypass productive struggle, that remove the ambiguity 
that builds expertise, that make it impossible to tell 
what you know vs. what the tool knows — those tools, 
as currently designed, have a specific cost.

The question isn't "AI or not." It's "how do we use these 
without eroding what makes the work meaningful."
```

## Likely Criticisms + Pre-Written Rebuttals

### "This is just burnout renamed"
```
AI Fatigue and burnout overlap but aren't the same. Burnout is about 
workload and emotional exhaustion. AI Fatigue is specifically about 
a change in the cognitive conditions that make engineering meaningful — 
skill atrophy, loss of authorship, identity disruption. 
The recovery paths are different.
```

### "Just learn to use the tools better"
```
This assumes the problem is individual skill. The structural issue is 
that AI tools, as currently designed, bypass the productive struggle 
that builds expertise — regardless of how skilled you are. 
Skill doesn't protect against this.
```

### "Productivity went up, not down"
```
Individual productivity often does go up. The concern is systemic: 
faster shipping with thinner understanding creates technical debt 
that compounds over time, and expertise that doesn't accumulate 
is expertise that erodes.
```
