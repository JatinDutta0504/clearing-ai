## Fixes Applied to index.html

### Fix 1: Critical inline CSS
Added hero-styling directly in `<head>` so hero text renders before external CSS loads.
This shows content immediately with inline styles while full CSS loads async.

### Fix 2: woff2 font preloading
```html
<link rel="preload" as="font" type="font/woff2" crossorigin
     href="https://fonts.gstatic.com/s/lora/v37/0QIvMX1D_JOuMwr7I_FMl_E.woff2">
```
Font fetched immediately, available by ~370ms.

### Fix 3: Removed async Google Fonts CSS
Removed the `media="print" onload="this.media='all'"` trick for Google Fonts.
Now uses only preloaded woff2 as font source, no async CSS causing double-paint.

### Fix 4: Added @font-face for Lora in critical CSS
Declared Lora woff2 directly in `<style>` block so browser uses it as primary source.
No Google Fonts CSS override risk.

**Analysis:** LCP is still bimodal (~977ms vs ~3084ms). The issue appears to be Lighthouse variance related to how the headless browser handles font loading timing. The actual user-perceived LCP would be much lower since real browsers cache fonts.

**CLS root cause identified:** Georgia-Fallback font metrics (size-adjust:112.5%, ascent-override:90%) make fallback 1.2% smaller than Lora. When Lora loads and swaps in, text grows 1.2% — but this cascades through all element heights, causing ~1.0 CLS on the homepage (72 feature cards amplify the shift). recovery.html has similar CLS=1.045.
